## 1. アプリ概要

本アプリは、家庭内の固定メンバー向けに、献立、買い物リスト、過去料理履歴、タグを一元管理するスマートフォン専用Railsアプリである。

主な利用フローは以下とする。

1. 今日から未来の献立を登録する。
2. 献立に必要な食材を買い物リストへ反映する。
3. 買い物リストで未購入・購入済みを管理する。
4. 過去になった献立を料理履歴へ自動移行する。
5. 過去料理をタグ・人物・キーワードで検索する。

写真機能、評価機能、朝食管理、AI献立提案、カロリー計算、過去料理から献立へ直接コピーする機能は実装しない。

---

## 2. 目的

- 今日から未来の昼食・夕食の献立を管理できるようにする。
- 献立から買い物リストを生成し、食材の買い忘れを減らす。
- 過去に食べた料理・外食を履歴として検索できるようにする。
- 料理タグ・人物タグで家庭内の食事記録を整理できるようにする。
- 一般ユーザーのデータをユーザー単位で分離する。
- 管理者がユーザーと全データを管理できるようにする。

## 3. 採用仕様

| 分類 | 採用仕様 |
|---|---|
| 食事区分 | 昼食、夕食のみ |
| 献立管理 | 今日から未来の献立を管理する |
| 自動移行 | 過去になった献立を料理履歴へ移行する |

## 5. ユーザー種別

### 5.1 一般ユーザー

一般ユーザーは以下を操作できる。

- 自分の献立管理
- 自分の過去料理閲覧・検索
- 自分の買い物管理
- 自分の料理タグ管理
- 自分の人物タグ管理
- 自分のプロフィール編集

一般ユーザーは他ユーザーのデータを閲覧・編集・削除できない。

### 5.2 管理者ユーザー

管理者ユーザーは以下を操作できる。

- 全ユーザー管理
- 全データ閲覧
- 全データ編集
- 全データ削除

管理者権限は強力なため、削除確認、管理者自身削除可否、最後の管理者削除可否、監査ログ要否は確認事項とする。

## 7. 画面別仕様

### 7.1 ログイン画面

#### 目的

登録済みユーザーがログインしてホーム画面へ遷移する。

#### 表示項目・入力項目

- メールアドレス
- パスワード
- ログイン状態保持
- ログインボタン
- 新規登録導線
- パスワード忘れ導線

#### バリデーション・例外

- メールアドレス必須
- メール形式
- パスワード必須
- 認証失敗時はログイン画面に留まり、エラーを表示する。

### 7.2 新規登録画面

#### 目的

新規ユーザーを登録する。

#### 入力項目

- メールアドレス
- パスワード
- パスワード確認

#### バリデーション

- メールアドレス必須
- メール形式
- メールアドレス重複不可
- パスワード必須
- パスワード6文字以上
- パスワード確認必須
- パスワード確認一致

### 7.3 ホーム画面

#### 目的

今日の献立を確認し、主要機能へ移動する。

#### 表示内容

- 今日の日付
- 今日の昼食
- 今日の夕食
- 各料理
- 料理タグ
- メモ
- 買い物リスト導線
- 献立作成導線
- 管理者ホーム導線（管理者のみ）

#### 重要仕様

- ホームは今日のみ表示する。
- 今日の献立は過去料理へ移行しない。
- 今日の昼食・夕食が未登録の場合は空状態を表示する。

### 7.4 献立一覧画面

#### 目的

今日以降の献立を日付順で一覧表示する。

#### 表示仕様

- 今日から未来のみ表示する。
- 日付順で表示する。
- 昼食・夕食を分けて表示する。
- 各時間帯に複数料理を表示できる。
- 空状態を表示する。

#### 重要仕様

- 同日同時間帯に複数料理を登録できる。
- 「同日同時間帯に複数献立可能」という表現は、データ構造上は「1献立枠に複数料理」とする案を推奨するが、確認事項とする。
- 今日の献立は一覧に残す。
- 昨日以前の献立は自動移行対象とし、献立一覧から消える。

### 7.5 献立作成画面

#### 目的

未来の献立を登録する。

#### 献立全体の入力項目

- 日付
- 昼食／夕食
- 人物タグ

#### 料理ごとの入力項目

- 料理名
- メモ
- 外食フラグ
- 料理タグ
- 必要食材

#### 人物タグ仕様

- 今回提示仕様では、人物タグは料理単位ではなく献立全体に付与する。
- 既存Markdownでは料理単位に人物タグを付与する仕様があるため、相反仕様として確認事項に記載する。

#### 必要食材仕様

- 料理ごとに必要食材を登録する。
- 各食材は買い物リストへ追加するかどうかのフラグを持つ。
- 買い物リスト追加チェックがONの食材のみ、買い物リストへ反映する。

#### バリデーション

- 日付必須
- 日付は原則として今日以降
- 食事区分必須
- 食事区分は昼食または夕食のみ
- 料理名必須
- 料理は1件以上必要

#### 保存仕様

- 献立、献立内料理、必要食材、タグ関連、人物タグ関連、買い物リスト追加は一括保存する。
- 途中で失敗した場合は全体ロールバックする案を推奨する。

### 7.6 買い物リスト画面

#### 目的

献立に必要な食材と手動追加した買い物を管理する。

#### 表示構造

買い物リストは以下の単位で表示する。

1. 日付
2. 昼食／夕食
3. 時系列順
4. 食材カード

表示例:

```text
5/22 夕食
  玉ねぎ
    使用料理: チキンカレー、オニオンスープ
  鶏肉
    使用料理: チキンカレー

手動追加
  牛乳
```

#### 食材統合仕様

- 同一日付・同一時間帯内で同一食材は統合する。
- 統合された食材には、使用料理一覧を表示する。
- 数量は表示しない。
- 統合の判定で前後空白、ひらがな・カタカナ、全角半角、大文字小文字をどう扱うかは確認事項とする。

#### 機能

- 未購入表示
- 購入済みチェック
- 未購入へ戻す
- 個別削除
- 購入済み一括削除
- 購入済み個別削除
- 購入済み30日経過で自動削除
- 手動追加

#### 手動追加仕様

- 手動追加分は献立日付・昼夕に紐づかない場合、「手動追加」グループに表示する。
- 手動追加分に日付・昼夕を指定できるかは確認事項とする。

### 7.7 過去料理一覧画面

#### 目的

過去料理を検索・閲覧する。

#### 検索対象

- 料理名
- メモ
- 料理タグ
- 人物タグ

#### 検索仕様

- 料理タグは複数選択可能。
- 人物タグは複数選択可能。
- 複数タグ選択時のAND/OR条件は確認事項とする。

#### 並び順

- 新しい日付順。
- 同日の昼夕順、作成日時順は確認事項とする。

#### 表示内容

- 日付
- 昼食／夕食
- 料理名
- メモ
- 外食フラグ
- 料理タグ
- 人物タグ

### 7.8 過去料理詳細画面

#### 表示内容

- 料理名
- 日付
- 昼食／夕食
- メモ
- 料理タグ
- 人物タグ
- 外食フラグ
- 編集ボタン
- 削除ボタン

#### 重要仕様

- 過去料理を手動編集・削除できる仕様として今回提示されている。
- 既存Markdownでは料理記録の手動作成・編集・削除が明確でないため、確認事項に記載する。

### 7.9 料理タグ管理画面

#### 機能

- 一覧
- 作成
- 編集
- 削除

#### 仕様

- 自由入力。
- 同一ユーザー内で重複不可。
- 削除時は確認モーダルを表示する。
- 削除したタグを過去料理・献立から外すか、無効化して履歴表示を残すかは確認事項とする。

### 7.10 人物タグ管理画面

#### 機能

- 一覧
- 作成
- 編集
- 削除
- デフォルト選択

#### 仕様

- 固定メンバー想定。
- 同一ユーザー内で重複不可。
- 複数選択可能。
- デフォルト選択の意味は確認事項とする。

### 7.11 管理者画面

#### 管理者ホーム

- ユーザー管理一覧への導線を表示する。
- 全データ管理への導線を表示する。

#### ユーザー管理一覧

- ユーザー一覧を表示する。
- ユーザー編集画面へ遷移できる。
- ユーザー削除操作を行える。

#### ユーザー編集画面

- メールアドレス
- 権限
- プロフィール項目

管理者が編集できるユーザー情報の範囲は確認事項とする。

## 8. データフロー

### 8.1 献立から過去料理

| 項目 | 内容 |
|---|---|
| 条件 | 昨日以前になった献立 |
| タイミング | ページ表示時 |
| 動作 | 過去料理へ移行し、献立一覧から削除または非表示 |
| 今日の献立 | 移行しない |

今回提示仕様の中に「今日以前になった献立」という表現があるが、同時に「今日の献立は移行しない」と明記されているため、本文では「昨日以前」を採用候補とし、矛盾・不足に記載する。

### 8.2 献立から買い物リスト

| 項目 | 内容 |
|---|---|
| 条件 | 買い物へ追加チェックON |
| 動作 | 未購入欄へ追加 |
| 同一食材 | 同一日付・同一時間帯内で統合する |
| 使用料理 | 統合食材に使用料理一覧を表示する |

### 8.3 外食

- 外食はサブ機能として扱う。
- 通常料理とほぼ同じ入力項目を持つ。
- 外食フラグを持つ。
- 必要食材を入力するかどうかは画面上可能だが、外食時は通常不要とする案を推奨する。

## 9. テーブル定義案

モデル名は今回提示仕様に合わせて `meal_plans` 系を候補とする。ただし既存Markdownでは `menu_plans` 系が使われているため、命名は確認事項とする。

| カラム | 型 | 制約 | 説明 |
|---|---|---|---|
| id | bigint | PK | ID |
| email | string | NOT NULL, UNIQUE | メールアドレス |
| encrypted_password | string | NOT NULL | 暗号化パスワード |
| role | integer | NOT NULL, default: 0 | 0:一般、1:管理者 |
| created_at | datetime | NOT NULL | 作成日時 |
| updated_at | datetime | NOT NULL | 更新日時 |

| カラム | 型 | 制約 | 説明 |
|---|---|---|---|
| id | bigint | PK | ID |

| plan_date | date | NOT NULL | 献立日 |
| meal_type | integer | NOT NULL | 0:昼食、1:夕食 |
| migrated_at | datetime | NULL | 過去料理へ移行済み日時。物理削除採用時は不要 |

推奨制約:

- `UNIQUE(user_id, plan_date, meal_type)` を付ける。ただし、同日同時間帯に複数献立枠を許す場合は付けない。

## 14. 自動処理

### 14.1 献立から過去料理への自動移行

- ページ表示時に実行する。
- 対象は昨日以前の献立とする案を推奨する。
- 今日の献立は移行しない。
- 移行後、献立一覧からは消す。
- 物理削除するか、`migrated_at` で非表示にするかは確認事項とする。
- 二重移行防止のため、移行元IDまたは別管理テーブルで一意性を担保する。

### 14.2 買い物リスト30日自動削除

- 購入済みから30日経過した項目を削除する。
- 実行タイミングは買い物リストページ表示時。
- 30日ちょうどを削除するか、30日超過で削除するかは確認事項とする。

## 16. 追加確認質問

### Q1. 買い物リストの同一食材統合について

既存要件では同名食材を重複表示、今回提示仕様では同一日付・同一時間帯内で統合となっています。

- A案：DBは食材1件ごとに保存し、表示時に日付・昼夕・食材名で集約する。
- B案：DB保存時点で統合して1件にまとめる。
- 推奨：A案。
- 質問：買い物リストの統合は表示上の集約でよいですか？

### Q2. 人物タグの付与単位について

既存要件では料理単位、今回提示仕様では献立全体に付与となっています。

- A案：献立全体に人物タグを付与する。
- B案：料理単位に人物タグを付与する。
- 推奨：A案。
- 質問：人物タグは献立全体に付与する仕様で確定してよいですか？

### Q3. 自動移行対象日について

今回提示仕様には「今日以前になった献立」と「今日の献立は移行しない」が混在しています。

- A案：昨日以前を移行対象にする。
- B案：今日以前を移行対象にする。
- 推奨：A案。
- 質問：今日の献立は残し、昨日以前のみ自動移行でよいですか？

### Q4. 献立自動移行後の元データについて

移行後に物理削除すると、移行元ID参照や二重移行防止が難しくなります。

- A案：物理削除する。
- B案：`migrated_at` を付けて非表示にする。
- 推奨：B案。
- 質問：献立一覧から消えれば、DB上は移行済みデータを保持してよいですか？

### Q5. 同日同時間帯のデータ構造について

「複数献立可能」と「複数料理可能」が混在しています。

- A案：1日・1食事区分に1つの献立枠を作り、その中に複数料理を入れる。
- B案：1日・1食事区分に複数の献立枠を作れる。
- 推奨：A案。
- 質問：同日同時間帯は1つの献立枠に複数料理で表現してよいですか？

### Q6. 過去料理の手動編集・削除について

過去料理詳細画面に編集・削除が含まれています。

- A案：過去料理は閲覧のみ。
- B案：過去料理を編集・削除できる。
- 推奨：B案。ただし権限と削除確認が必要。
- 質問：過去料理の編集・削除を実装対象に含めますか？

### Q7. 手動追加の買い物項目について

手動追加分は「手動追加」グループ表示とされています。

- A案：日付・昼夕を持たず「手動追加」にまとめる。
- B案：手動追加時にも日付・昼夕を任意指定できる。
- 推奨：A案。
- 質問：手動追加の買い物項目は日付・昼夕なしでよいですか？

### Q8. 管理者の削除権限について

管理者は全データ編集削除可能とされています。

- A案：全データを即時編集・削除可能。
- B案：削除は確認画面、削除対象件数表示、最後の管理者削除禁止を入れる。
- 推奨：B案。
- 質問：管理者削除には二段階確認と最後の管理者削除禁止を入れてよいですか？

### Q9. パスワード忘れ機能について

ログイン画面にパスワード忘れ導線がありますが、機能要件は未定義です。

- A案：MVPでは導線を非表示にする。
- B案：パスワードリセット機能を実装する。
- 推奨：A案。
- 質問：パスワード忘れ機能は初期実装に含めますか？

### Q10. プロフィール編集項目について

一般ユーザーは自分の情報編集ができるとされています。

- A案：メールアドレスとパスワードのみ編集可能。
- B案：表示名などプロフィール項目も追加する。
- 推奨：A案。
- 質問：プロフィール編集はメールアドレス・パスワード変更だけでよいですか？
