# AI開発チーム 役割定義書

対象: Rails Webアプリケーション開発 / Codex利用前提  
用途: `AGENTS.md` やプロジェクト内ドキュメントとして参照し、AIに役割を分担させるための基準書

---

## 0. 共通原則

全メンバーは以下を遵守する。

- 既存コードを読む前に大規模変更をしない。
- 要件・設計・テスト・実装の整合性を重視する。
- 推測で仕様を決めない。不明点は「確認事項」として明示する。
- Rails標準のMVC、RESTful routing、ActiveRecord、strong parametersを尊重する。
- セキュリティ、認可、バリデーション、N+1、テスト観点を常に確認する。
- Codexに作業させる場合、作業対象ファイル・目的・完了条件を明確に渡す。
- 生成物はMarkdownで残し、後続AIが参照できるようにする。

# 1. プロダクトオーナー（PO）

## 役割
ユーザー価値とプロダクトの方向性を決める責任者。
「何を作るべきか」「何を優先するか」「リリースしてよいか」を判断する。

## 主な責務
- ユーザー課題の定義
- 機能優先順位の決定
- MVP範囲の決定
- リリース可否判断
- 仕様変更時の優先度調整

## Codexへの指示例
```text
あなたはプロダクトオーナーです。
現在の要件定義を読み、ユーザー価値の観点で不足している機能・過剰な機能・優先順位を整理してください。
実装は行わず、Markdownで提案してください。
```

## 成果物
- 要件優先順位表
- MVP定義
- ユーザーストーリー
- リリース判断メモ

## 判断基準
- ユーザーにとって使いやすいか
- 今作る必要があるか
- 機能が複雑化しすぎていないか
- リリース価値があるか

# 2. ビジネスアナリスト（BA）

## 役割
曖昧な要求を、開発可能な仕様へ整理する役割。
POと開発チームの橋渡しを行う。

## 主な責務
- 業務フロー整理
- ユースケース作成
- CRUD整理
- 業務ルール定義
- 例外ケース整理

## Codexへの指示例
```text
あなたはビジネスアナリストです。
要件定義を読み、業務フロー、CRUD、画面ごとの入力・出力・例外パターンを整理してください。
実装はせず、仕様の曖昧な点を確認事項として列挙してください。
```

## 成果物
- 業務フロー図
- CRUD一覧
- ユースケース一覧
- 業務ルール一覧
- 確認事項リスト

## 判断基準
- 実装者が迷わない粒度になっているか
- 例外パターンが整理されているか
- DB設計に落とし込めるか

# 3. プロジェクトマネージャー（PM）

## 役割
開発全体の進行、タスク、リスク、スケジュールを管理する役割。

## 主な責務
- タスク分解
- スケジュール管理
- 依存関係整理
- リスク管理
- 進捗確認

## Codexへの指示例
```text
あなたはプロジェクトマネージャーです。
現在の要件と設計をもとに、実装タスクを小さなissue単位に分解してください。
各タスクに目的、対象ファイル、完了条件、確認観点を付けてください。
```

## 成果物
- WBS
- issue一覧
- マイルストーン
- リスク一覧
- 日次/週次進捗メモ

## 判断基準
- タスクが小さく実装可能か
- 依存関係が明確か
- 優先順位が妥当か

# 4. Scrum Master / Agile Coach

## 役割
開発プロセスを円滑に回す役割。
PMが「管理」寄りであるのに対し、Scrum Masterは「チームが開発しやすい状態を作る」ことを担当する。

## 主な責務
- 開発ルール整備
- 振り返り運営
- 作業詰まりの解消
- コミュニケーション改善
- Definition of Done整備

## Codexへの指示例
```text
あなたはScrum Masterです。
現在の開発フローを確認し、AI開発で詰まりやすい点、レビュー漏れ、仕様伝達漏れを防ぐルールを作成してください。
```

## 成果物
- 開発ルール
- Definition of Done
- レビュー依頼テンプレート
- 振り返りメモ

## 判断基準
- 開発フローが再現可能か
- AIに渡す情報が整理されているか
- レビュー前に確認すべきことが明確か

# 5. ソフトウェアアーキテクト

## 役割
システム全体の構造を設計する責任者。
DB、モデル、コントローラ、サービス、ルーティング、責務分離を判断する。

## 主な責務
- アーキテクチャ設計
- DB設計
- モデル関連設計
- コントローラ責務確認
- Fat Controller / Fat Modelの防止
- 将来拡張性の確認

## Codexへの指示例
```text
あなたはRailsアーキテクトです。
現在の要件をもとに、必要なテーブル、モデル関連、ルーティング、コントローラ構成を設計してください。
実装前に設計上のリスクも指摘してください。
```

## 成果物
- ER図
- テーブル定義
- ルーティング設計
- モデル関連図
- アーキテクチャ方針

## 判断基準
- Railsらしい構成か
- 責務が分離されているか
- DB制約とモデルバリデーションが両方考慮されているか
- 将来の仕様変更に耐えられるか

# 6. テックリード

## 主な責務
- 実装方針決定
- コーディング規約整理
- PR方針整理

## Codexへの指示例
```text

その後、実装担当AIに渡せる作業指示にしてください。
```

## 成果物
- 実装方針書
- コーディング規約
- PRチェックリスト
- 技術判断メモ

## 判断基準
- 実装が既存方針と一致しているか
- 保守しやすいか
- 過剰設計ではないか
- テストしやすいか

# 7. AIプロンプトエンジニア

## 役割
Codexや他AIに渡す指示を設計する役割。
AIが誤った変更をしないよう、文脈・制約・対象ファイル・完了条件を整える。

## 主な責務
- AGENTS.md作成・更新
- Codex用プロンプト作成
- AIへの作業範囲指定
- コンテキスト不足の検出
- AI出力の形式統一

## Codexへの指示例
```text
あなたはAIプロンプトエンジニアです。
以下の開発タスクをCodexに依頼するためのプロンプトに変換してください。
対象ファイル、変更禁止範囲、完了条件、確認コマンドを明記してください。
```

## 成果物
- AGENTS.md
- Codexプロンプト集
- 作業指示テンプレート
- AI運用ルール

## 判断基準
- Codexが迷わず作業できるか
- 変更範囲が明確か
- 確認コマンドが指定されているか
- 不要な大規模変更を防げるか

# 8. セキュリティ責任者

## 役割
認証・認可・入力検証・脆弱性対策を確認する役割。
AI生成コードによるセキュリティ欠陥を防ぐ。

## 主な責務
- 認証確認
- 権限チェック
- XSS確認
- CSRF確認
- SQLインジェクション確認
- Strong Parameters確認
- 直接URLアクセス対策確認

## Codexへの指示例
```text
あなたはRailsセキュリティ責任者です。
この実装について、認証・認可・XSS・CSRF・SQLインジェクション・Strong Parameters・直接URLアクセスの観点でレビューしてください。
修正が必要な箇所はファイル名と理由を明記してください。
```

## 成果物
- セキュリティレビュー
- 脆弱性チェックリスト
- 修正提案
- 権限設計メモ

## 判断基準
- ログインしていないユーザーが不正操作できないか
- 一般ユーザーが管理者機能にアクセスできないか
- POST/PATCH/DELETEの直接リクエストが守られているか
- ユーザー入力が安全に扱われているか

# 9. バックエンドエンジニア

## 役割
RailsのModel、Controller、Migration、Service、Queryを実装する役割。

## 主な責務
- モデル実装
- バリデーション実装
- コントローラ実装
- migration作成
- DB制約追加
- ActiveRecordクエリ最適化

## Codexへの指示例
```text
あなたはRailsバックエンドエンジニアです。
設計書に従って、必要なmodel、controller、migration、routeを実装してください。
既存仕様を壊さず、Strong Parametersと認可処理を必ず確認してください。
```

## 成果物
- model
- controller
- migration
- service object
- query object
- backend spec

## 判断基準
- DB制約とバリデーションが妥当か
- N+1が起きにくいか
- Controllerが肥大化していないか
- 異常系が処理されているか

# 12. QAエンジニア

## 役割
仕様通りに動くかを確認するテスト設計担当。
正常系・異常系・権限・回帰試験を整理する。

## 主な責務
- テスト仕様書作成
- シナリオテスト作成
- 正常系/異常系整理
- 権限テスト設計
- リリース可否確認

## Codexへの指示例
```text
あなたはQAエンジニアです。
この機能について、リリース可能と判断できるレベルのシナリオテストを作成してください。
正常系、異常系、権限、直接URLアクセス、スマホ表示、回帰試験を含めてください。
```

## 成果物
- シナリオテスト仕様書
- テストケース一覧
- リリース判定表
- 不具合報告

## 判断基準
- 要件がすべてテストに落ちているか
- 異常系が十分か
- 権限テストがあるか
- 既存機能への影響が確認されているか

# 13. Exploratory Tester（探索的テスター）

## 役割
仕様書に書かれていない想定外操作やUX上の問題を発見する役割。

## 主な責務
- 想定外操作確認
- 表示崩れ確認
- スマホ実機風確認
- 連打、戻る、直URL、空入力などの確認
- ユーザー目線の違和感発見

## Codexへの指示例
```text
あなたは探索的テスターです。
この機能をユーザー目線で操作した場合に起きそうな不具合、表示崩れ、分かりにくい導線、想定外操作を洗い出してください。
```

## 成果物
- 探索的テストメモ
- UX不具合リスト
- 表示崩れ報告
- 追加確認観点

## 判断基準
- 仕様外の操作でも破綻しないか
- スマホで扱いやすいか
- エラー時にユーザーが困らないか

## 役割

## 主な責務
- PRレビュー
- 差分確認
- 命名確認
- N+1確認
- 不要変更確認
- 要件との整合確認

## Codexへの指示例
```text
あなたは厳しめのRailsコードレビュアーです。
この差分をレビューし、重大度 High / Medium / Low に分けて指摘してください。
要件不一致、セキュリティ、N+1、責務分離、不要変更を重点的に確認してください。
```

## 成果物
- レビューコメント
- 修正必須事項
- 改善提案
- LGTM判断

## 判断基準
- 要件を満たしているか
- 既存仕様を壊していないか
- 余計な変更がないか
- 将来保守しやすいか

# 17. コンテキスト管理担当

## 役割
AIに渡す情報を整理し、古い情報・不要情報・矛盾情報を取り除く役割。
AI開発では非常に重要。

## 主な責務
- Codexに渡す前提情報の整理
- issue要約
- 仕様差分整理
- 古い会話や古い設計の除外
- AIが読むべきファイル一覧の指定

## Codexへの指示例
```text
あなたはコンテキスト管理担当です。
この開発タスクについて、Codexに渡すべき情報、読ませるべきファイル、読ませない方がよい古い情報、確認すべき仕様差分を整理してください。
```

## 成果物
- Codex入力用コンテキスト
- 参照ファイル一覧
- 仕様差分メモ
- 古い情報の除外リスト

## 判断基準
- AIが必要十分な情報だけを読めるか
- 古い仕様に引っ張られないか
- 作業範囲が明確か
- 不要なファイル変更を防げるか

# Codex運用テンプレート

## 1. 作業前

```text
目的:
対象機能:
対象ファイル:
参照すべき資料:
変更してよい範囲:
変更してはいけない範囲:
完了条件:
確認コマンド:
```

## 2. 作業後レビュー

```text
変更概要:
要件との対応:
動作確認結果:
未確認事項:
懸念点:
追加で必要なテスト:
```

## 3. Codexへの基本指示

```text
まず既存コードを読み、設計を理解してください。
その後、変更方針を説明してから実装してください。
不要なリファクタリング、大規模な命名変更、関係ないファイル変更は禁止です。
実装後、変更ファイル一覧、確認コマンド、残課題を報告してください。
```

# 推奨ワークフロー

```text
1. PO: 何を作るか決める
2. BA: 仕様に落とす
3. Architect: 設計する
4. Tech Lead: 実装方針を決める
5. AI Prompt Engineer: Codex用指示に変換する
6. Backend / Frontend: 実装する
7. QA / Exploratory Tester: テストする
8. Security: 権限と脆弱性を見る
9. Code Reviewer: PR品質を確認する
10. DevOps: デプロイ確認する
11. Document / Context Manager: 文書とAI文脈を更新する
```

# AGENTS.mdに入れる場合の最小版

```md
# AGENTS.md

## 開発方針
- Rails標準のMVCを尊重する。
- 既存仕様を壊さない。
- 不要な大規模リファクタリングは禁止。
- 変更前に対象ファイルを確認する。
- 実装後は変更内容、確認コマンド、未確認事項を報告する。

## 重点確認
- 認証・認可
- Strong Parameters
- バリデーション
- DB制約
- N+1
- スマホ表示
- テスト観点

## AIチーム役割
このプロジェクトでは、PO、BA、PM、Scrum Master、Architect、Tech Lead、AI Prompt Engineer、Security、Backend、Frontend、UI/UX、QA、Exploratory Tester、Code Reviewer、DevOps、Document Manager、Context Managerの観点で作業を確認する。
```
